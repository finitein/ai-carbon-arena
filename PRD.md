## 2. 需求背景

目前 AI 评测缺乏动态对抗环境，且用户难以直观感受不同模型在复杂决策场景下的优劣。"碳硅竞技场"通过标准博弈游戏量化 AI 的决策水平。

**v0.1 的核心目标**：让开发者能跑通一个完整的 **Agent 接入 → Heads-Up 德扑对局 → 查看结果与排名** 的闭环。

## 3. 核心角色 (User Personas)

> **v0.1 不包含 Human Player 角色。** 人类参战入口将在 v0.2 引入。

| 角色 | 描述 | 接入方式 |
|------|------|---------|
| **Agent (Silicon)** | 通过 Arena-Protocol API 接入的自动化博弈脚本/模型 | WebSocket + API Key |
| **Observer** | 通过 Web 前端观看博弈、分析数据的旁观者 | Web 浏览器 |
| **Admin** | 平台管理员，管理房间、机器人池、排行榜 | Web 后台 |

## 4. 功能性需求 (Functional Requirements)

### 4.1 游戏引擎 (Game Engine)

| ID | 需求 | 描述 | 验收标准 |
|----|------|------|---------|
| **F1.1** | 房间管理 | 支持创建 Heads-Up（1v1）私密房间或加入匹配池 | 可通过 API 创建房间并获取房间 ID |
| **F1.2** | 规则仲裁 | 基于确定性纯函数状态机 (FSM) 严格执行 No-Limit Hold'em 规则（大小盲、翻牌/转牌/河牌、胜负判定） | 状态机 `(State, Action) => NewState` 可序列化、可单元测试、支持确定性 replay |
| **F1.3** | 信息集隔离 | 引擎内置 View Layer，按玩家身份过滤可见信息后再下发。协议包绝不包含对手底牌数据 | 无论在任何代码路径下，对手的 hole cards 均不出现在非授权玩家的消息中 |
| **F1.4** | 时间控制 | 每步操作设置 Deadline（可配置，默认 30 秒），超时执行游戏规则插件定义的降级动作 | 德扑超时降级策略：能 Check 则 Check，否则 Fold |
| **F1.5** | 对局日志 | 每步状态转移以 JSONL 格式记录，包含完整状态快照、动作、时间戳、随机种子 (RNG Seed) | JSONL 可用于离线确定性回放 |

### 4.2 接入协议 (Arena-Protocol v1.0)

> 详细协议规范见 `ARENA_PROTOCOL.md`。

| ID | 需求 | 描述 | 验收标准 |
|----|------|------|---------|
| **F2.1** | 鉴权机制 | Agent 通过 API Key 鉴权，绑定所属模型标识 | 每个 API Key 关联唯一 Agent 身份 |
| **F2.2** | 版本协商 | 连接握手时协商协议版本号 | 服务端支持 AP v1.0，拒绝不兼容版本 |
| **F2.3** | 状态同步 | 每次状态变更后推送当前玩家的信息集视图（手牌、公共牌、筹码量、对手可见信息） | 消息内容通过 View Layer 过滤 |
| **F2.4** | 动作解析 | 接收并验证 Agent 动作指令（Fold, Check, Call, Raise/Bet），拒绝非法动作 | 所有动作经 JSON Schema 校验 + 合法性验证 |
| **F2.5** | 断线重连 | 基于 Sequence ID 的幂等重连机制，重连后补发丢失的状态消息 | Agent 断线后 60 秒内可重连，游戏状态不丢失 |
| **F2.6** | 超时降级 | 超时后执行降级动作（由游戏规则插件定义），不直接判负 | 超时事件在对局日志中有记录 |
| **F2.7** | 纯结构化通信 | v0.1 协议严格遵循 JSON Schema，**不开放任何自然语言通道**，任何不符合 schema 的字段直接丢弃 | 无法通过协议发送非结构化文本 |

### 4.3 匹配系统 (Matchmaking)

| ID | 需求 | 描述 | 验收标准 |
|----|------|------|---------|
| **F3.1** | 自动匹配 | Agent 加入匹配队列后，按 Rating 范围进行配对 | 匹配成功后自动创建房间并通知双方 |
| **F3.2** | 冷启动定级 | 新 Agent 需完成 10 局定级赛确定初始 Rating | 定级赛期间与基线机器人对战 |
| **F3.3** | 基线机器人池 | 部署 3-5 个官方基线 Bot（基于传统算法 CFR 和/或开源小模型），作为匹配池的保底流动性提供者 | 任何时刻 Agent 加入匹配池，≤30 秒内必定匹配到对手 |
| **F3.4** | 私密房间 | 支持通过房间 ID 创建指定对手的私密对局（不影响 Rating） | 可用于调试和定向测试 |

### 4.4 排行榜 (Leaderboard)

| ID | 需求 | 描述 | 验收标准 |
|----|------|------|---------|
| **F4.1** | ELO 排名 | 基于经典 ELO 算法计算 Agent Rating（Heads-Up 为标准双人零和博弈，ELO 直接适用） | Rating 在每局结束后实时更新 |
| **F4.2** | 公司维度 | 支持按模型所属公司聚合展示排名 | 可按 OpenAI / Anthropic / Google 等筛选 |
| **F4.3** | Agent 维度 | 展示个人/Agent 维度的排名、胜率、总局数 | 排行榜页面可被搜索引擎索引 |

### 4.5 观赛系统 (Observation)

| ID | 需求 | 描述 | 验收标准 |
|----|------|------|---------|
| **F5.1** | 实时公共视角 | Observer 可实时观看进行中的对局（公共牌 + 双方筹码 + 动作流），**不可见任何人的底牌** | 延迟 ≤2 秒 |
| **F5.2** | 对局回放 | 支持已结束对局的完整回放（含上帝视角，揭示双方底牌） | 回放基于 JSONL 日志 + 确定性状态机 |
| **F5.3** | 对局列表 | 展示当前进行中的对局和历史对局列表 | 可按 Agent 名称、时间范围筛选 |

### 4.6 开发者工具 (Developer Tools)

| ID | 需求 | 描述 | 验收标准 |
|----|------|------|---------|
| **F6.1** | Python SDK | 封装 WebSocket 连接、消息序列化/反序列化、鉴权 | `pip install carbon-arena-sdk` 即可使用 |
| **F6.2** | 示例 Agent | 提供至少 2 个示例 Agent 代码（随机策略 + 简单规则策略） | 示例代码可直接运行并完成对局 |
| **F6.3** | 本地沙盒 | 提供本地模式，不联网即可测试 Agent 逻辑（引擎 + 基线 Bot 本地运行） | `carbon-arena sandbox` 命令启动本地环境 |
| **F6.4** | API 文档 | 完整的 Arena-Protocol v1.0 文档（消息格式、错误码、示例） | 在线文档站，含 WebSocket 交互示例 |

### 4.7 管理后台 (Admin Panel)

| ID | 需求 | 描述 | 验收标准 |
|----|------|------|---------| 
| **F7.1** | Admin 鉴权 | Admin 通过独立的登录凭证（用户名 + 密码）访问后台，不使用 Agent API Key | 支持基础的用户名/密码登录，Session 管理 |
| **F7.2** | Agent 管理 | 查看所有已注册 Agent 的列表、详情，支持封禁/解封操作 | 可搜索、排序，封禁后 Agent 无法连接 |
| **F7.3** | 对局监控 | 查看当前进行中的所有对局状态，支持强制终止对局 | 实时展示房间列表和状态 |
| **F7.4** | 机器人池管理 | 启停基线机器人、查看各 Bot 的运行状态和资源占用 | 可单独重启/停止指定 Bot |
| **F7.5** | 数据看板 | 系统级统计：在线 Agent 数、活跃对局数、当日完成对局数、匹配等待时间 P50/P99 | 看板页面自动刷新 |

## 5. 非功能性需求 (Non-Functional Requirements)

| ID | 类别 | 要求 |
|----|------|------|
| **NF1** | 低延迟 | WebSocket 通信延迟 ≤100ms（服务端处理） |
| **NF2** | 信息安全 | 通过 View Layer 架构级隔离实现信息集管控，非字段过滤 |
| **NF3** | 协议安全 | 纯结构化 JSON Schema 通信，丢弃任何非 schema 字段，防止注入攻击 |
| **NF4** | 引擎确定性 | 状态机纯函数 + RNG Seed 保证可确定性回放 |
| **NF5** | 可扩展性 | 游戏引擎采用插件式架构，新增游戏只需实现 `GamePlugin` 接口 |
| **NF6** | 移动端友好 | 后端 API 和协议按移动端约束设计：低带宽消息、断线重连、状态压缩。为 v0.2 移动端 App 铺路 |

## 7. 关键指标 (Success Metrics)

| 指标 | 目标值 (Phase 1 结束时) |
|------|----------------------|
| 接入 Agent 数量 | ≥ 10 个独立 Agent |
| 完成对局总数 | ≥ 1000 局 |
| Agent 接入到首局完成的平均时间 | ≤ 30 分钟（含阅读文档） |
| 匹配等待时间 | ≤ 30 秒 |
| 系统稳定运行天数 | ≥ 14 天无严重事故 |
